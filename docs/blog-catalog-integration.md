# ТЗ: подгрузка скиллов на блог (страница «Skills Library»)

Техническое задание для реализации страниц **списка скиллов** и **детальной
страницы скилла** на блоге (`evgenyosipchuk.com`) на основе данных из репозитория
`agent-skills`.

## 1. Источник данных и URL

Единственный источник — сгенерированный файл **`catalog.json`** в корне репозитория.
Он автоматически пересобирается CI при каждом мердже в `main` и коммитится обратно,
поэтому всегда доступен по HTTPS:

```
https://raw.githubusercontent.com/<owner>/agent-skills/main/catalog.json
```

Рекомендуется брать `<owner>` из самого `catalog.json` (`library.repo`), а не
хардкодить. Для кеширования на CDN можно использовать commit-pinned URL или
jsDelivr (`https://cdn.jsdelivr.net/gh/<owner>/agent-skills@main/catalog.json`).

`registry.json` (рядом) — это **манифест установщика**, не для блога. Блог читает
**только** `catalog.json`.

## 2. Чего делать НЕ нужно: парсить GitHub

**Не** обходите дерево репозитория через GitHub API и **не** парсите `SKILL.md` /
`catalog/*.yaml` на стороне блога. Причины:

- репозиторий уже распарсил всё это в CI и отдал готовый `catalog.json`;
- парсинг дерева упирается в rate-limit GitHub API;
- это дублирование логики (frontmatter, YAML, валидация, вычисление дат) и почти
  гарантированный дрейф между блогом и репозиторием.

Единственное «живое» обращение к GitHub, которое допустимо, — опциональный бейдж
«issues · N open» (см. §6). Всё остальное — из `catalog.json`.

## 3. Стратегия загрузки

- **Build-time (рекомендуется):** тянуть `catalog.json` на этапе сборки (SSG/ISR),
  рендерить статические страницы. Никаких клиентских запросов к GitHub, нет
  rate-limit.
- **Кеш / ревалидация:** при build-time — ревалидация раз в N минут/часов (ISR) или
  ребилд по вебхуку. При commit-pinned URL — кеш можно держать «вечно».
- **Отказоустойчивость:** при неудачной загрузке отдавать последний успешный
  снапшот (last-good), не падать.
- **`schema_version`:** перед рендером сверить `catalog.json.schema_version`. На
  текущий момент `"1.0"`. Неизвестную мажорную версию — логировать и отдавать
  last-good, а не рендерить вслепую.

## 4. Схема `catalog.json`

Авторитетная JSON-схема лежит рядом: `catalog.schema.json` (сгенерирована из
моделей). Ниже — краткая выжимка.

Верхний уровень:

| Поле | Тип | Описание |
|------|-----|----------|
| `schema_version` | string | версия формата (`"1.0"`) |
| `library` | object | блок библиотеки (см. ниже) |
| `skills` | array&lt;CatalogSkill&gt; | список скиллов |

`library`:

| Поле | Тип | Описание |
|------|-----|----------|
| `name` | string | имя библиотеки (`askill-library`) |
| `repo` | string | URL репозитория (источник `slug` и ссылок) |
| `generated_at` | string (ISO 8601) | когда сгенерирован каталог |
| `commit` | string | SHA коммита, к которому привязан каталог |

`CatalogSkill` (элемент `skills`):

| Поле | Тип | Обяз. | Описание |
|------|-----|:----:|----------|
| `name` | string | да | kebab-case идентификатор/имя |
| `version` | string | да | semver `MAJOR.MINOR.PATCH` |
| `description` | string | да | короткий «dek» (1 строка, 10–1024 симв.) |
| `description_long` | string \| null | нет | длинный триггер-текст (для блока «When it fires») |
| `path` | string | да | путь к папке скилла, напр. `skills/re-anchor` |
| `entry` | string | да | файл-вход, по умолчанию `SKILL.md` |
| `tags` | string[] | да* | теги (kebab-case, ≤10); может быть `[]` |
| `compatible_agents` | string[] | да | включает `claude-code` |
| `license` | string \| null | нет | напр. `MIT` |
| `updated_at` | string (ISO 8601) \| null | нет | дата последнего коммита папки скилла |
| `when` | string \| null | нет | абзац «когда срабатывает» |
| `highlights` | string[] | нет | пункты «что делает» |
| `example` | object \| null | нет | трейс-пример (см. §7) |

\* `tags` всегда присутствует как массив (возможно пустой).

> Опциональные поля (`description_long`, `when`, `highlights`, `example`,
> `license`, `updated_at`) при отсутствии **опускаются** в JSON (генерация идёт с
> `exclude_none`). Блог должен корректно работать, когда их нет.

## 5. Маппинг полей на элементы дизайна

| Элемент дизайна | Страница | Источник |
|-----------------|----------|----------|
| имя скилла | список + деталь | `name` |
| `skills/{slug}/` под именем | список | `path` (или `slug`, см. §6) |
| краткое описание (`dek`) | список + хедер детали | `description` |
| теги + фильтр + счётчики | список + деталь | `tags` (фильтр/счётчики считаются на клиенте) |
| «updated …» | строка списка + сайдбар | `updated_at` |
| `github.com/{REPO} · N skills · MIT` | хедер списка | `library.repo`, `skills.length`, `license` |
| ссылка на GitHub-папку | деталь | производное (§6) |
| ссылка `SKILL.md` | деталь | производное (§6) |
| «When it fires» (абзац) | деталь | `when` (фолбэк — `description_long`) |
| блок `name: … / description: …` | деталь | `name` + `description_long` |
| «What it does» (буллеты) | деталь | `highlights` |
| блок «Example» | деталь | `example` (§7) |
| сайдбар metadata (path/updated/license/tags) | деталь | `path` / `updated_at` / `license` / `tags` |
| «pairs well with» | деталь | производное (§6) |
| «issues · N open» | деталь | живой GitHub API (опц., §6) |

## 6. Производные значения (считать на блоге, не в репозитории)

- **`slug`** = последний сегмент `path` (`skills/re-anchor` → `re-anchor`).
- **GitHub-папка** = `{library.repo}/tree/main/{path}` (или закрепить на `commit`).
- **raw `SKILL.md`** = из `library.repo` + `path` + `entry`.
- **список тегов и счётчики** для фильтра = агрегировать по `skills[].tags`
  (`all` = общее число скиллов).
- **«pairs well with»** = другие скиллы, у которых пересекается ≥1 тег с текущим.
- **«issues · N open»** — опционально, живой запрос к GitHub REST
  (`/repos/{owner}/{repo}/issues?...`). Помнить про rate-limit: кешировать или не
  показывать.

## 7. Рендер блока Example

`example` — объект:

```json
{
  "title": "long-form EN→DE with a glossary",
  "turns": 4,
  "tokens": 210,
  "caption": "…",
  "trace_url": null,
  "input":  [ { "kind": "user", "text": "…" }, { "kind": "tool", "text": "…" } ],
  "output": [ { "kind": "anchor", "text": "…" }, { "kind": "assistant", "text": "…" } ]
}
```

`input` и `output` — упорядоченные списки реплик. Поле `kind` определяет стиль
«пузыря» (см. дизайн `skill-detail.jsx`, компонент `MessageBubble`):

| `kind` | Значение | Стиль |
|--------|----------|-------|
| `user` | реплика пользователя | обычный фон |
| `tool` | вызов инструмента | «sunken» фон, моноширинный |
| `tool_out` | результат инструмента | «sunken» фон, моноширинный |
| `assistant` | ответ ассистента | обычный фон |
| `anchor` | **момент срабатывания скилла** | акцентный фон/рамка (подпись «skill») |

`title`, `turns`, `tokens`, `caption`, `trace_url` — для шапки/подвала блока; любое
из них может отсутствовать. Если у скилла нет `example` / `highlights` / `when` —
соответствующую секцию просто не рендерить (graceful degradation).

## 8. Совместимость на будущее

- Новые скиллы появляются в `catalog.json` автоматически при следующей генерации —
  никаких правок на блоге не требуется.
- Парсер блога должен **игнорировать неизвестные поля**, а не падать на них (формат
  может расширяться в рамках той же мажорной `schema_version`).
- Не полагаться на порядок скиллов сверх того, что они отсортированы по `name`.
